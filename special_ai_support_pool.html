안녕하세요. 오늘은 특별 강의로 "Claude Code를 활용한 AI 기반 XenServer 기술지원 Pool 구축"에 대해 이야기하겠습니다. 핵심 질문은: "매번 수천 줄의 로그를 수동으로 분석하고, 같은 장애에 반복 대응하는 것이 최선인가?" 전반부 35분은 강의, 후반부 20분은 Hands-on 실습입니다.

왜 AI 기술지원인가?

AS-IS (현재의 한계)

  수천 줄 xensource.log 수동 분석 → 시간 소모
  동일 장애 반복 대응 → 노하우 단절
  엔지니어 퇴사 시 지식 유실
  영어 기술문서 해석 부담

TO-BE (AI 지원 후)

  로그 자동 분석 → 원인 즉시 진단
  대응 이력 축적 → 재활용 가능
  AI에 지식 내재화 → 팀 전체 공유
  다국어 기술문서 즉시 해석/요약
  단순한 도구 도입이 아니라, 기술지원 방식의 패러다임 전환
XenServer 장애 발생 시 xensource.log를 수동으로 분석하면 최소 30분~2시간이 걸립니다. AI가 이것을 30초로 줄일 수 있다면?

  핵심 차별점: CLAUDE.md에 XenServer 노하우를 적으면 → AI가 그 지식을 참조하여 진단
Claude Code는 Anthropic의 AI CLI 도구입니다. CLAUDE.md에 HA Self-Fence 대응 절차를 적어두면, 로그 분석 시 그 절차를 참조합니다.

기술지원 Pool 아키텍처

①
Layer 1: CLAUDE.md 지식 베이스
장애 유형 카탈로그 | 진단 명령어 | SOP
②
Layer 2: Claude Code AI Engine
로그 분석 | 패턴 매칭 | 진단 | 보고서 생성
③
Layer 3: MCP 서버 연동
Filesystem | SSH | Database | Web API
지식 + AI + 연결 = AI 기술지원 Pool 완성
3개 레이어가 연결되면 "XenServer 호스트에서 로그를 읽고, AI가 분석하고, 보고서를 작성"하는 전체 파이프라인이 완성됩니다.

Step 1

CLAUDE.md로 XenServer 지식 베이스 구축

1
장애 유형 카탈로그 작성
HA Self-Fence, Storage Disconnect, VM Hang — 교육에서 배운 장애 유형을 그대로 문서화
2
진단 명령어 레퍼런스
xe host-list, xe task-list, cat /var/log/xensource.log — 정상/비정상 결과 판별 기준
3
체크리스트 & SOP 내장
HA 구성 전 체크리스트, 장애 대응 Do's & Don'ts, Vinchin 백업 검증 절차
  완벽하게 작성할 필요 없음 — 장애 대응할 때마다 하나씩 추가하면 지식이 축적됨
첫 번째 단계는 CLAUDE.md 파일을 만드는 것입니다. 완벽할 필요 없습니다. 장애를 대응할 때마다 하나씩 추가하면 됩니다.

Step 2

MCP 서버로 실시간 시스템 연동

Filesystem MCP 이것 하나로 80%
xensource.log, SMlog 직접 읽기/분석 — 처음 시작할 때 가장 효과적
SSH MCP
원격 XenServer에 xe 명령 실행 & 결과 자동 분석
Database MCP
Vinchin 백업 이력 DB 조회 — 성공률, 실패 로그, RPO 준수 확인
Web API MCP
Zabbix/PRTG 모니터링 연동 — 실시간 서버 상태, 알림 이력 수집
모든 MCP를 한꺼번에 구성할 필요 없습니다. Filesystem MCP 하나만 연결해도 AI 분석의 80%가 가능합니다.

Step 3

자동화 워크플로우 구성

1
로그 수집
MCP로 XenServer/Vinchin 로그 자동 수집
2
AI 분석
패턴 매칭, 이상 탐지, 원인 분석 수행
3
진단 보고서
장애 원인, 영향 범위, 권장 조치를 보고서로 자동 생성
4
대응 실행
검증된 스크립트 실행 또는 수동 가이드 · 최종 결정은 반드시 엔지니어
"사람이 판단하고, AI가 실행한다" — AI가 스스로 시스템을 변경하지 않음
핵심 원칙: AI는 분석과 보고서 작성을 담당하고, 최종 실행 결정은 반드시 엔지니어가 합니다.

# XenServer 기술지원 지식 베이스

## 장애 유형 카탈로그

### HA Self-Fence
- 증상: 호스트 자동 재부팅
- 원인: Network/Storage Heartbeat 손실
- 대응:
  1. xensource.log에서 fencing 타임스탬프
  2. Network Bond 상태 확인 (xe pif-list)
  3. Storage Heartbeat 디스크 확인

### 핵심 CLI 명령어
- xe host-list: 풀 내 호스트 상태
- xe task-list: 실행 중 작업 확인
- xe vm-list: VM 상태 일괄 조회

## Do's & Don'ts
- DO: 로그 우선 확보
- DON'T: 원인 없이 강제 재부팅

Before (CLAUDE.md 없이)

컨텍스트 부족

우리 팀의 SOP에 맞춘 맞춤형 진단
Do's & Don'ts 자동 참조

실습 3 · 6분

XenServer 로그 샘플 AI 분석

실제 xensource.log 샘플을 Claude Code에 전달하여 AI 분석 결과를 확인합니다
# Step 1: 분석 요청
> "sample-xensource.log 파일을 분석해줘.
  1. 타임라인을 구성하고
  2. 주요 에러/경고를 추출하고
  3. 근본 원인(Root Cause)을 추정하고
  4. 권장 조치 사항을 제시해줘"
# Step 2: 보고서 생성
> "지금 분석한 내용을 고객에게 전달할 수 있는
  장애 분석 보고서 형식으로 작성해줘"
  AI의 분석 결과는 반드시 엔지니어가 검증해야 합니다. AI는 보조 도구입니다.
보고서 형식으로 변환하면 에스컬레이션이나 고객 보고에 즉시 활용할 수 있습니다. 반드시 엔지니어가 검증하세요.
