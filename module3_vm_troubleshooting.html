VM 생성/템플릿 & XenTools 필수성

  "VM 생성은 단순 설치가 아니라, 검증된 '운영 정책'과 '설정'이 포함된 Master Template을 복제하는 과정"
구분ISO 직접 설치Template 복제
소요 시간30분~1시간1분 내외
일관성낮음 (Human Error)완벽 일치
관리 용이성개별 관리 필요중앙 표준화
  운영 리스크: 템플릿 설정 오류 시 → 해당 템플릿으로 생성된 수백 대 VM에서 동일 장애 발생
VM 생성은 반드시 Template을 통해 하세요. ISO 직접 설치는 시간 낭비이고 일관성을 보장할 수 없습니다.

상태XenTools 미설치 XenTools 설치됨

XenTools 없이 VM을 운영하는 것은 블랙박스를 다루는 것과 같습니다. 백업, HA, DR 등 모든 고급 기능의 전제 조건이 XenTools입니다.

유형Crash-consistentLive Snapshot

  "Snapshot은 백업이 아니다" — 특정 시점으로 돌아가기 위한 '임시 마커'일 뿐. 원본 VHD 손상 시 Snapshot도 함께 소실
작업 후 반드시 삭제

Snapshot은 '사진 한 장'이 아니라 '디스크 체인의 분기점'입니다. 시간이 지나면 체인이 길어지고 성능이 떨어집니다. 진짜 백업은 Vinchin 같은 전용 솔루션으로 해야 합니다.

Section 3-3 · 10분

주요 장애 유형

유형 1 Host Down / Pool Coordinator 장애
원인: 하드웨어 장애, 네트워크 고립, 스토리지 경로 유실 → 커널 패닉, Dom0 리소스 고갈
Supporter Down: 해당 호스트 VM만 중단
Coordinator Down: 기존 VM 유지, 관리 작업 불가
유형 2 Storage Disconnect
원인: 물리 단절, LUN Mapping 오류, 컨트롤러 Failover 실패, 네트워크 과부하
iSCSI: TCP Timeout 민감 · FC: Fast Fail · NFS: Stale File Handle
유형 3 VM Hang / Boot Failure
Hang: Storage I/O Latency (가장 흔함), vCPU/Memory 경합, OS 내부 문제
Boot: No Bootable Device, PV Driver 충돌, Corrupted VDI
핵심: VM 자체 문제보다 외부 요인(Snapshot Coalesce, Storage Path, XenTools 불일치)인 경우가 많음
  절대 금지: 스토리지 복구 전 VM 강제 재부팅 → 파일시스템 손상 및 디스크 체인 파손 위험
장애 분류를 외울 필요는 없습니다. 핵심은 '이 증상이 VM 내부 문제인가, 아니면 인프라 문제인가?'를 먼저 구분하는 것입니다.

장애 대응: Do's and Don'ts

절대 금지 (Don'ts)

  원인 파악 없이 Host 강제 재부팅
→ 로그 유실, 디스크/DB 체인 손상
  Pool Coordinator 무작정 변경
→ Split-brain, 메타데이터 꼬임
  스토리지 복구 전 VM 강제 시작
→ I/O 오류 증폭

권장 조치 (Do's)

  xensource.log, SMlog 우선 확보
  XenCenter 및 iDRAC/ILO 접속 테스트
  multipath -ll 경로 상태 확인
  top, xentop Dom0 부하 확인

재부팅 판단 기준

"재부팅은 문제 해결(Fix)이 아니라 상태 초기화(Reset)입니다"

원인이 명확히 규명? 로그 사라져도 OK? 복구가 다운타임보다 급함?
장애 상황에서는 아무것도 하지 않는 것이 가장 좋은 대응일 수도 있습니다. 로그를 먼저 확보하세요.

모듈 3 Q&A

Q. Snapshot은 얼마나 자주 찍어도 되나요?
A. 백업용이 아닌 작업용으로만. 24~72시간 내 삭제 원칙

Q. 장애 발생 시 가장 먼저 할 일은?
A. xensource.log 확보 → 장애 범위 파악(VM? Host? Storage?) → 강제 재부팅 금지

Q. Dom0에 패키지를 설치해도 되나요?
A. 절대 금지. 문서화되지 않은 변경은 Upgrade/Operation 문제 유발
