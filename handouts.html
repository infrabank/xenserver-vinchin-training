HA vs Live Migration 판단 기준

XenServer 환경의 안정적 운영을 위한 전략적 판단 기준 · 모듈 2 참고
H1

다음 조건 중 2개 이상 해당되면 HA(자동화)보다 Live Migration(수동 통제)이 더 안전합니다.

#운영 환경 조건기술적 의미 및 위험 요소해당
1호스트 2대 구성HA 동작 시 리소스 부족, Split-brain 중재 어려움 (구조적 불안)☐
2관리 네트워크 단일 구성네트워크 일시 단절을 호스트 장애로 오탐지 → 정상 VM 재기동 위험☐
3Shared Storage 불안정Heartbeat Disk 접근 실패 → HA 오동작(Fencing) 및 VM 강제 종료☐
4VM 수가 적은 환경소규모 환경에서는 HA 자동화 이점 < 오동작 리스크☐
5재기동에 민감한 서비스HA는 장애 복구 시 필연적으로 VM 재부팅(서비스 중단) 동반☐
6운영자 상시 대응 가능기계적 자동화보다 엔지니어 판단에 의한 수동 조치가 더 안전☐
7계획 점검 위주 환경돌발 장애 대응보다 계획된 유지보수(무중단 이동)가 운영 핵심☐
HA (High Availability)
자동 재기동 수행 · 오탐지 시 리스크 · 서비스 일시 중단
Live Migration
수동 통제 · 서비스 중단 없음 · 예측 가능한 운영
"자동화보다 통제가 중요한 환경에서는 Live Migration이 더 안전합니다"

HA 안전 구성 체크리스트

XenServer 8.x · 점검일자: ______________ · 점검자: ______________ · 모듈 2 참고
H5

1. 네트워크 구성 * 가장 중요

  • Management Network NIC Bond 구성 (LACP/Active-Passive)
  • 서로 다른 물리 스위치에 업링크 연결 (Switch Redundancy)
  • 관리망(Management)과 VM 트래픽 분리 (VLAN/NIC)
  • 스위치 STP / LACP 설정 검증 완료
  • 관리망 패킷 드롭(Packet Drop) 발생 이력 없음

2. 스토리지 / HA SR

  • HA SR은 Shared Storage (iSCSI, FC, NFS)
  • 모든 Host에서 동일 SR 접근 가능 확인
  • 스토리지 다중 경로(Multipath) 구성 (단일 경로 금지)
  • MPIO 정상 동작 확인 (경로 단절 테스트 완료)
  • 스토리지 컨트롤러 장애 시 Failover 테스트 완료

3. 용량 설계 (N-1 기준)

  • Host 1대 장애 시 남은 Host가 모든 Protected VM 수용 가능
  • Failover 시 CPU/Memory 리소스 여유분 확보 (Overcommit 주의)
  • 부트 스톰(Boot Storm) 발생 시 스토리지 I/O 대역폭 여유 존재

4. VM 정책 설정

  • HA Protected VM 최소화 (핵심 서비스만 보호 설정)
  • Restart Priority 구분 적용 (High: 핵심, Medium: 일반, Low: 기타)
  • Anti-Affinity(분산 배치) 필요한 이중화 VM 설정 확인

5. 운영 정책

  • HA 동작 중 수동 재기동/개입 금지 정책 전파
  • 장애 발생 시 물리 Host 상태 및 HA 로그 선 확인 절차 준수
  • 정기적인 로그 수집 및 상태 분석 절차 정립
모든 항목이 YES여야 안전합니다. 하나라도 NO가 있다면, HA는 보호 장치가 아니라 시스템 전체 장애를 유발할 수 있는 위험 요소입니다.
